# 西雅图建筑能耗预测与节能改造优先级评估：端到端数据挖掘工程手册

> 适用课题：城市建筑能耗预测与节能改造优先级评估  
> 本地数据快照：Seattle Building Energy Benchmarking Data，2015—2024  
> 本手册依据题目要求、`gpt.md` 中的讨论历史、当前工作区两份 CSV 的实际审计结果整理。  
> 本项目输出的是“优先开展能源审计与改造可行性评估的候选名单”，不是未经工程审计即可执行的投资决策。

## 0. 先读这一页：项目最终要做什么

本课题最稳妥的研究定义是：

> 使用建筑用途、规模、年代、空间位置和年度等预测时可获得的属性，估计建筑在某报告年度的合理 `SiteEUI` 基准；再结合模型残差、官方同类建筑基准、连续多年表现、理论超额能耗量和碳排放，识别异常高耗能建筑并形成节能改造初筛优先级。

这里的“预测”是数据驱动的能耗基准回归，不是严格意义上的“用历史能耗预测下一年能耗”。主模型不使用上一年 `SiteEUI`，也不使用当年总能耗、能源分项或排放等可反推出目标的字段。

完整主线固定为：

```text
原始数据留档与版本记录
    ↓
字段类型转换、数据质量审计、跨年度一致性检查
    ↓
探索性数据分析（EDA）
    ↓
无泄漏特征工程 + 按完整年份划分
    ↓
基线模型 + Ridge/ElasticNet + Random Forest + XGBoost + LightGBM
    ↓
时间外测试、分组误差、SHAP 与失败案例分析
    ↓
时间外残差 + 官方同类基准 + 持续性识别高耗能异常
    ↓
“建筑—年份”汇总为“一栋建筑一行”
    ↓
多指标优先级评分、权重敏感性和排序稳定性分析
    ↓
对高优先级候选进行聚类画像和地图展示
    ↓
候选清单、数据核查清单、图表、模型和结题报告
```

完成项目时至少要交付：

1. 可重复运行的 Python 工程；
2. 数据质量报告和疑似错误记录清单；
3. EDA 图表；
4. 基线与多模型对比、2023/2024 独立测试结果；
5. 最优模型及 SHAP 解释；
6. 年度异常高耗能清单；
7. 建筑级 Top-20、Top-50、Top-100 改造初筛名单；
8. 权重敏感性、排序稳定性和聚类画像；
9. 完整实验报告。

## 3. 第一步：安全读取和字段类型统一

### 3.1 为什么不能直接依赖 `pandas` 自动类型

CSV 中大量数值带千位逗号，如 `"7,113,958"`，因此 `PropertyGFATotal`、`SiteEUI`、`SiteEnergyUse`、排放和能源分项等会被读成字符串。若不处理，描述统计、相关性和模型都会出错。

### 3.2 数值解析模板

将下面逻辑写入 `src/io.py`，并保留原列名；不要一开始就删除原字段。

```python
from pathlib import Path
import pandas as pd

NUMERIC_WITH_COMMAS = [
    "PropertyGFATotal",
    "PropertyGFABuildings",
    "PropertyGFAParking",
    "SelfReportGFATotal",
    "SelfReportGFABuildings",
    "SelfReportParking",
    "SiteEUIWN(kBtu/sf)",
    "SiteEUI(kBtu/sf)",
    "SiteEnergyUse(kBtu)",
    "SiteEnergyUseWN(kBtu)",
    "SourceEUIWN(kBtu/sf)",
    "SourceEUI(kBtu/sf)",
    "LargestPropertyUseTypeGFA",
    "SecondLargestPropertyUseTypeGFA",
    "ThirdLargestPropertyUseTypeGFA",
    "Electricity(kWh)",
    "Electricity(kBtu)",
    "SteamUse(kBtu)",
    "NaturalGas(therms)",
    "NaturalGas(kBtu)",
    "TotalGHGEmissions",
    "GHGEmissionsIntensity",
]


def parse_numeric(series: pd.Series) -> pd.Series:
    return pd.to_numeric(
        series.astype("string").str.replace(",", "", regex=False).str.strip(),
        errors="coerce",
    )


def read_main(path: str | Path) -> pd.DataFrame:
    df = pd.read_csv(path, low_memory=False)
    for col in NUMERIC_WITH_COMMAS:
        if col in df.columns:
            df[col] = parse_numeric(df[col])
    df["OSEBuildingID"] = df["OSEBuildingID"].astype("string")
    df["ZipCode"] = df["ZipCode"].astype("string")
    df["CouncilDistrictCode"] = df["CouncilDistrictCode"].astype("string")
    return df
```

标识符和行政编码即使由数字组成也不是连续变量。`OSEBuildingID`、`TaxParcelIdentificationNumber`、邮编和区议会编码不得被模型理解为大小有意义的数值。

### 3.3 读取后立即执行的断言

## 4. 第二步：数据质量审计与双通道处理

### 4.1 不要把数据错误和业务高耗能混为一类

建立两套互不替代的标记：

```text
DataQualityFlag：缺失、单位、口径、逻辑或上报疑点
HighEnergyFlag：数据可信，但能耗显著高于合理基准
```

极端 `SiteEUI` 可能是真实特殊建筑，也可能是单位或上报错误。不得仅凭“高于 99% 分位数”自动删除，也不得把它直接列为改造优先对象。

### 4.2 必做审计表

对全体、各年度、合规/不合规分别统计：

- 行数、唯一建筑数、进入/退出建筑数；
- 字段类型、缺失数、缺失率、唯一值数；
- `SiteEUI` 有效率；
- 合规状态和拆除状态；
- 各类别频数及类别名称随年份的变化；
- `OSEBuildingID + DataYear` 唯一性；
- 数值范围和关键分位数；
- 逻辑异常和算术一致性。

当前 `SiteEUI` 缺失率在 2021、2022 年明显升高，分别约 12.33% 和 9.48%。合规记录缺失率约 1.59%，不合规记录约 33.81%。报告中必须说明该缺失并非完全随机。

### 4.3 逻辑和算术检查

至少实现：

```python
df["DQ_DuplicateIDYear"] = df.duplicated(
    ["OSEBuildingID", "DataYear"], keep=False
)
df["DQ_InvalidYearBuilt"] = (
    df["YearBuilt"].isna() | (df["YearBuilt"] > df["DataYear"])
)
df["DQ_FloorZeroOrMissing"] = (
    df["NumberofFloors"].isna() | (df["NumberofFloors"] <= 0)
)
df["DQ_InvalidArea"] = (
    df["PropertyGFABuildings"].isna()
    | (df["PropertyGFABuildings"] <= 0)
)
df["DQ_MissingOrInvalidTarget"] = (
    df["SiteEUI(kBtu/sf)"].isna()
    | (df["SiteEUI(kBtu/sf)"] <= 0)
)
df["DQ_NotCompliant"] = df["ComplianceStatus"].ne("Compliant")
df["DQ_Demolished"] = df["Demolished"].fillna(False)

gfa_diff = (
    df["PropertyGFATotal"]
    - df["PropertyGFABuildings"]
    - df["PropertyGFAParking"]
).abs()
df["DQ_GFAIdentityFailed"] = gfa_diff > 0.01
```

还要检查：

- 纬度、经度是否落在合理的西雅图范围；
- `SiteEnergyUse` 与能源分项合计是否严重矛盾；
- 同一建筑相邻年份的面积、用途和能耗是否突变；
- `SiteEUI × 面积` 与 `SiteEnergyUse` 的口径是否相容；
- 极端记录在相邻年份是否同样极端；
- `ComplianceIssue` 是否说明异常原因。

### 4.4 EUI 面积口径

本地检查显示，`SiteEnergyUse / SiteEUI` 与 `SelfReportGFABuildings` 的一致性通常最好，但不能假定所有年度和建筑都采用同一个面积字段。计算“理论超额能耗量”时优先定义：

```python
df["EUIReferenceArea"] = (
    df["SiteEnergyUse(kBtu)"] / df["SiteEUI(kBtu/sf)"]
)
df.loc[~df["EUIReferenceArea"].gt(0), "EUIReferenceArea"] = pd.NA
df["EUIReferenceArea"] = df["EUIReferenceArea"].fillna(
    df["SelfReportGFABuildings"]
).fillna(df["PropertyGFABuildings"])
```

该字段只用于结果解释和排序，不能进入 `SiteEUI` 预测模型，因为其首选值由当年目标和当年总能耗共同得到。

### 4.5 极端记录的审计规则

创建 `outputs/tables/data_quality_review.csv`，至少纳入满足任一条件的记录：

- `SiteEUI` 位于“对标类型 × 年份”99.5% 分位数以上；
- `SiteEUI` 是同一建筑最近三年中位数的 5 倍以上或不足 1/5；
- 推断的 `EUIReferenceArea` 与可用上报面积相差超过 50%；
- `YearBuilt > DataYear`、面积非正、层数非正；
- 能源分项合计与总量严重不一致；
- 同年用途/面积发生无法解释的突变。

这些是“人工复核触发器”，不是自动删除规则。最终记录一列：

```text
DQDecision ∈ {keep, exclude_from_model, audit_only}
DQReason
ReviewerNote
```

### 4.6 两组建模样本

建立两个实验层：

1. `high_quality`：未拆除、合规、目标有效、面积和用途有效、未被人工判定为错误；
2. `all_valid`：未拆除、目标和核心属性有效，同时保留数据质量标记。

主结果优先报告 `high_quality`，并用 `all_valid` 做敏感性对照。这样既不盲目删除极端值，也能验证数据质量筛选是否改变结论。

## 5. 第三步：构造可连接官方区间表的 `BenchmarkType`

### 5.1 这是一个强制步骤

主表 `BuildingType` 只有 8 个粗类别，例如 `NonResidential`、`Multifamily LR (1-4)`；绩效区间表 `BuildingType` 是 23 个 OSE 对标类别，例如 `Hotel`、`Laboratory`、`Large Office`。当前数据中两者精确类别交集为 0，直接按 `BuildingType + DataYear` 合并的匹配率是 0%。

直接按 `LargestPropertyUseType` 合并也只覆盖约 25.6% 的有效 `SiteEUI` 记录。因此必须先构造统一的 `BenchmarkType`。

### 5.2 映射规则

映射以 `EPAPropertyType` 为主：

| 主表条件 | `BenchmarkType` |
|---|---|
| `Multifamily LR (1-4)` | `Low-Rise Multifamily` |
| `Multifamily MR (5-9)` | `Mid-Rise Multifamily` |
| `Multifamily HR (10+)` | `High-Rise Multifamily` |
| `Office` 且建筑面积 < 100,000 sf | `Small- and Mid-Sized Office` |
| `Office` 且建筑面积 ≥ 100,000 sf | `Large Office` |
| `Hospital (General Medical & Surgical)` | `Hospital` |
| 已与官方 23 类同名的类型 | 保留同名 |
| 其他未列入类型 | `Other` |

办公面积优先使用不含停车的 `PropertyGFABuildings`。官方历史说明将 1—4 层、5—9 层、10 层及以上分别定义为低层、中层、高层多户住宅，并以 100,000 平方英尺区分大型办公楼。

实现模板：

```python
BENCHMARK_TYPES = {
    "College/University",
    "Distribution Center",
    "High-Rise Multifamily",
    "Hospital",
    "Hotel",
    "K-12 School",
    "Laboratory",
    "Large Office",
    "Low-Rise Multifamily",
    "Medical Office",
    "Mid-Rise Multifamily",
    "Mixed Use Property",
    "Non-Refrigerated Warehouse",
    "Other",
    "Refrigerated Warehouse",
    "Residence Hall/Dormitory",
    "Restaurant",
    "Retail Store",
    "Self-Storage Facility",
    "Senior Living Community",
    "Small- and Mid-Sized Office",
    "Supermarket/Grocery Store",
    "Worship Facility",
}

TYPE_RENAME = {
    "Multifamily LR (1-4)": "Low-Rise Multifamily",
    "Multifamily MR (5-9)": "Mid-Rise Multifamily",
    "Multifamily HR (10+)": "High-Rise Multifamily",
    "Hospital (General Medical & Surgical)": "Hospital",
}


def make_benchmark_type(row) -> str:
    epa_type = row["EPAPropertyType"]
    if epa_type in TYPE_RENAME:
        return TYPE_RENAME[epa_type]
    if epa_type == "Office":
        return (
            "Large Office"
            if row["PropertyGFABuildings"] >= 100_000
            else "Small- and Mid-Sized Office"
        )
    if epa_type in BENCHMARK_TYPES:
        return epa_type
    return "Other"
```

### 5.3 合并和验收

```python
df["BenchmarkType"] = df.apply(make_benchmark_type, axis=1)
df = df.merge(
    benchmark,
    left_on=["BenchmarkType", "DataYear"],
    right_on=["BuildingType", "DataYear"],
    how="left",
    validate="many_to_one",
    suffixes=("", "_Benchmark"),
)
```

必须检查：

- `BenchmarkType` 是否只包含官方 23 类；
- 合并后 `Median_EUI`、`Percentile_75th_EUI` 是否 100% 有值；

## 7. 第五步：确定预测特征并封死目标泄漏

### 7.1 预测目标

```text
y = SiteEUI(kBtu/sf)
训练内部可变换为 y_log = log1p(y)
所有最终指标必须还原到原始 kBtu/sf 尺度计算
```

### 7.2 主模型可用特征

类别特征建议：

```text
BuildingType
EPAPropertyType
LargestPropertyUseType
Neighborhood
CouncilDistrictCode
ZipCode（可选，先检查稀疏性）
```

连续特征及构造方式：

```python
df["BuildingAge"] = df["DataYear"] - df["YearBuilt"]
df.loc[df["BuildingAge"] < 0, "BuildingAge"] = pd.NA

df["LogGFABuildings"] = np.log1p(df["PropertyGFABuildings"].clip(lower=0))
df["LogGFAParking"] = np.log1p(df["PropertyGFAParking"].clip(lower=0))
df["FloorMissingOrZero"] = df["NumberofFloors"].fillna(0).le(0).astype(int)

safe_floors = df["NumberofFloors"].where(df["NumberofFloors"] > 0)
safe_buildings = df["NumberofBuildings"].where(df["NumberofBuildings"] > 0)

df["AreaPerFloor"] = df["PropertyGFABuildings"] / safe_floors
df["AreaPerBuilding"] = df["PropertyGFABuildings"] / safe_buildings
df["ParkingRatio"] = (
    df["PropertyGFAParking"] / df["PropertyGFATotal"].where(
        df["PropertyGFATotal"] > 0
    )
)
df["PrimaryUseRatio"] = (
    df["LargestPropertyUseTypeGFA"]
    / df["SelfReportGFABuildings"].where(df["SelfReportGFABuildings"] > 0)
)
df["HasSecondUse"] = df["SecondLargestPropertyUseType"].notna().astype(int)
df["HasThirdUse"] = df["ThirdLargestPropertyUseType"].notna().astype(int)
df["UseCount"] = 1 + df["HasSecondUse"] + df["HasThirdUse"]
df["GFADeviation"] = (
    (df["SelfReportGFATotal"] - df["PropertyGFATotal"]).abs()
    / df["PropertyGFATotal"].where(df["PropertyGFATotal"] > 0)
)
```

再加入：`DataYear`、`YearBuilt` 或 `BuildingAge`（二者同时使用前检查共线性）、`NumberofFloors`、`NumberofBuildings`、纬度、经度以及数据异常指示变量。

### 7.3 主模型禁止字段

| 禁止字段 | 原因 |
|---|---|
| `OSEBuildingID`、名称、地址、地块号 | 标识/记忆建筑，不代表可泛化属性 |
| `SiteEUIWN` | 与目标同源 |
| `SiteEnergyUse`、`SiteEnergyUseWN` | 可结合面积近似还原目标 |
| `SourceEUI`、`SourceEUIWN` | 同期能耗派生量 |
| 电力、天然气、蒸汽用量 | 当年实际能源消费 |
| `TotalGHGEmissions`、`GHGEmissionsIntensity` | 由当年能源消费派生 |
| `ENERGYSTARScore` | 基于当年能源绩效，不是纯静态输入 |
| 官方 `Median_EUI`、四分位数 | 只能做外部基准，不进入主预测特征 |
| `EUIReferenceArea` | 其首选值由目标和总能耗共同得到 |

`ComplianceStatus` 和 `ComplianceIssue` 用于筛选和数据质量分析，不进入“合理能耗基准”的主模型。可在附加实验中加入，并明确该实验回答的是不同问题。

### 7.4 类别编码

不能对无序用途使用普通 `LabelEncoder`。统一使用：

```python
OneHotEncoder(
    handle_unknown="ignore",
    min_frequency=20,
    sparse_output=True,
)
```

数值变量用训练集拟合的中位数填补；类别变量用 `Unknown` 填补。所有预处理必须封装进 `Pipeline`，并且只在训练折拟合，严禁先对全数据拟合填补、缩放或类别合并器。

阶段验收：建立一份明确的 `feature_manifest.json`，其中包含目标、允许特征、禁止特征、每个构造特征的公式和使用阶段。

## 8. 第六步：时间划分与两个泛化问题

### 8.1 主评估：完整年份切分

不得随机打乱全部 34,699 行后划分。固定主实验：

| 集合 | 年份 | 用途 |
|---|---|---|
| 训练 | 2015—2021 | 拟合候选模型 |
| 验证 | 2022 | 调参、选模型、定阈值 |
| 测试 1 | 2023 | 锁定后第一次时间外测试 |
| 测试 2 | 2024 | 锁定后第二次时间外测试 |

测试集不得参与特征筛选、模型选择、调参和异常阈值确定。2023 和 2024 必须分别报告，也可补充合并结果。

### 8.2 训练内部扩展窗口

建议折次：

```text
训练 2015—2017 → 验证 2018
训练 2015—2018 → 验证 2019
训练 2015—2019 → 验证 2020
训练 2015—2020 → 验证 2021
训练 2015—2021 → 验证 2022
```

实现：

```python
def expanding_year_splits(years, first_valid_year=2018, last_valid_year=2022):
    years = np.asarray(years)
    splits = []
    for valid_year in range(first_valid_year, last_valid_year + 1):
        train_idx = np.flatnonzero(years < valid_year)
        valid_idx = np.flatnonzero(years == valid_year)
        splits.append((train_idx, valid_idx))
    return splits
```

传入搜索器前先 `reset_index(drop=True)`，确保这里的整数索引与 `X`、`y` 行位置一致。

### 8.3 时间外残差必须来自未见过该年度的模型

异常识别不能使用训练集内拟合残差。为 2018—2024 年逐年生成滚动时间外预测：

```text
预测 2018：只训练 2015—2017
预测 2019：只训练 2015—2018
...
预测 2024：只训练 2015—2023
```

把这些预测保存为 `out_of_time_predictions.parquet`。最近三年异常和最终排序只使用这种时间外预测。模型超参数应在测试解封前确定，避免每年根据当年结果反复调参。

### 8.4 辅助评估：未见建筑泛化

时间切分允许同一建筑的早年记录进入训练集，这符合“已知建筑未来年度基准”的应用场景，但不等价于对新建筑的泛化。

另做一次按 `OSEBuildingID` 的组划分：

```text
80% 建筑 ID 训练，20% 建筑 ID 测试；测试 ID 不得出现在训练中。
```

报告应区分：

- 时间外、可能见过同一建筑：年度基准更新能力；
- 从未见过建筑 ID：新建筑泛化能力。

## 9. 第七步：基线、模型训练与调参

### 9.1 先建立三个基线

所有中位数只能从训练数据计算，未知类别回退到全体训练中位数。

1. 全体训练样本中位数；
2. `BenchmarkType` 训练中位数；
3. `LargestPropertyUseType` 训练中位数。

没有基线就不能说明复杂模型是否真正带来价值。

### 9.2 四类主模型

至少比较：

- Ridge 或 ElasticNet：可解释的线性参照；
- Random Forest：Bagging 树模型；
- XGBoost：Boosting 树模型；
- LightGBM：高效 Boosting 树模型。

不需要为了数量再堆大量模型。重点是不同模型家族、正确验证和误差分析。

### 9.3 目标变换

`SiteEUI` 极度右偏，推荐将模型包在：

```python
from sklearn.compose import TransformedTargetRegressor

reg = TransformedTargetRegressor(
    regressor=pipeline,
    func=np.log1p,
    inverse_func=np.expm1,
)
```

必须同时做一个原始目标实验，判断 `log1p` 是否降低 MAE、改善尾部表现。最终预测负值统一裁为 0，但要报告裁剪比例；若比例明显不为零，应重新检查模型。

### 9.4 预处理流水线

```python
numeric_pipe = Pipeline([
    ("imputer", SimpleImputer(strategy="median", add_indicator=True)),
    ("scaler", StandardScaler()),
])

categorical_pipe = Pipeline([
    ("imputer", SimpleImputer(strategy="most_frequent")),
    ("onehot", OneHotEncoder(
        handle_unknown="ignore",
        min_frequency=20,
        sparse_output=True,
    )),
])

preprocess = ColumnTransformer([
    ("num", numeric_pipe, numeric_features),
    ("cat", categorical_pipe, categorical_features),
])
```

树模型一般不需要缩放数值变量，可为线性模型和树模型分别建立预处理器。不要为了复用代码机械地给所有模型使用完全相同的缩放流程。

### 9.5 有限而合理的参数搜索

推荐 `RandomizedSearchCV` 或自写年度循环，使用上一节的完整年份折，主评分 `neg_mean_absolute_error`。示例范围：

```text
Ridge: alpha = logspace(-3, 3)
ElasticNet: alpha、l1_ratio
RandomForest: n_estimators、max_depth、min_samples_leaf、max_features
XGBoost: n_estimators、learning_rate、max_depth、min_child_weight、subsample、colsample_bytree
LightGBM: n_estimators、learning_rate、num_leaves、min_child_samples、feature_fraction
```

控制搜索预算，例如每模型 20—40 组组合。选择依据按优先级排序：

1. 多折平均 MAE；
2. 各年份 MAE 的稳定性；
3. 2022 验证表现；
4. 小样本用途和高能耗尾部的误差；
5. 训练成本和可解释性。

不能只挑某一个年份 R² 最高的模型。

阶段输出：

```text
models/best_model.joblib
outputs/tables/cv_results.csv
outputs/tables/baseline_comparison.csv
outputs/reports/model_card.md
```

模型卡至少写明数据版本、样本筛选、特征、禁止字段、切分、参数、适用范围和限制。

## 10. 第八步：评价、置信区间和误差诊断

### 10.1 指标

在原始 `kBtu/sf` 尺度计算：

```python
metrics = {
    "MAE": mean_absolute_error(y_true, y_pred),
    "RMSE": mean_squared_error(y_true, y_pred) ** 0.5,
    "R2": r2_score(y_true, y_pred),
    "MedAE": median_absolute_error(y_true, y_pred),
}
```

MAE 是主指标；RMSE 观察大误差；R² 表示总体方差解释；MedAE 表示典型建筑误差。MAPE 不作为主指标，因为目标接近 0 时分母会放大误差。

与最佳基线比较：

```text
MAE Improvement = (MAE_baseline - MAE_model) / MAE_baseline
```

不要把 `R² > 0.75` 设为项目成败的刚性门槛。更合理的成功标准是模型在两个时间外测试年均稳定优于基线，并能支撑可解释的异常筛选。

### 10.2 必须分组报告

- 2023、2024 独立结果；
- `BenchmarkType`；
- 主要用途；
- 面积分组；
- 建筑年龄分组；
- 合规/不合规（附加样本实验）；
- 目标值普通区间与最高 10% 尾部。

样本量小于 30 的组不单独下强结论，合并为 `Small-n` 或仅列出并标注不稳定。

### 10.3 相关样本的置信区间

同一建筑跨年重复出现，普通逐行 bootstrap 会低估不确定性。以 `OSEBuildingID` 为块进行 bootstrap，重复抽取建筑 ID 后计算 MAE/RMSE，报告 95% 区间。

### 10.4 必做图和失败案例

- 实际值—预测值散点图，带 `y=x`；
- 残差直方图和残差—预测值图；
- 各年度和用途 MAE；
- 校准曲线或预测分箱内实际均值；
- 误差最大的 20 条记录；
- 同一建筑连续多年系统性高估/低估；
- 极高 `SiteEUI` 对 RMSE 的影响；
- `high_quality` 与 `all_valid` 结果对照。

如果模型总体良好但某类建筑长期低估，异常名单会不公平地把该类建筑推高，必须先修正模型或在同类残差内判断。

## 12. 第十步：年度异常高耗能识别

### 12.1 使用三类互补证据

第一类：时间外模型残差。

```text
Residual(i,t) = ActualEUI(i,t) - PredictedEUI(i,t)
ResidualRatio(i,t) = Residual(i,t) / max(PredictedEUI(i,t), ε)
```

第二类：官方同类年度基准。

```text
BenchmarkExcessRatio(i,t)
    = (ActualEUI(i,t) - MedianEUI(type,t)) / MedianEUI(type,t)
```

并判断实际值是否超过官方 `Percentile_75th_EUI`。

第三类：最近三年持续性。

```text
Persistence = 异常年份数 / 有效年份数
```

### 12.2 残差异常阈值

核心方法使用“同一 `BenchmarkType × DataYear` 内残差前 10%”：

```python
pred["ResidualPercentile"] = pred.groupby(
    ["BenchmarkType", "DataYear"]
)["Residual"].rank(pct=True, method="average")

pred["ModelHighFlag"] = (
    (pred["Residual"] > 0)
    & (pred["ResidualPercentile"] >= 0.90)
)
```

小样本类型处理：

- 当年组样本数 ≥ 20：使用当年同类分位数；
- 当年组样本数 < 20：使用该类型最近三个时间外年度的合并残差分布；
- 合并后仍不足 20：标记 `SmallGroupFlag`，不得仅凭分位数进入核心名单。

可补充鲁棒 z 分数：

```text
RobustZ = (Residual - median_group) / (1.4826 × MAD_group)
```

`RobustZ > 2` 作为辅助证据。若时间允许，再用 2022 校准残差构造单侧预测上界；实际值超过预测上界可作为第三种模型证据。

### 12.3 核心年度异常定义

建议同时满足：

```text
ModelHighFlag = True
OfficialHighFlag = ActualEUI > Percentile_75th_EUI
BenchmarkExcessRatio > 0
DataQualityEligible = True
Demolished = False
```

`Residual > 0` 本身绝不能定义异常，因为约一半样本可能为正。

输出分成两个名单：

1. `annual_high_energy_candidates.csv`：数据可信且高耗能；
2. `annual_data_audit_candidates.csv`：数值极端或数据质量不足，应先核查。

## 13. 第十一步：从年度记录汇总到建筑级

最终决策单位必须是一栋建筑一行。以 2024 为当前状态，以 2022—2024 表示持续性。

每个 `OSEBuildingID` 计算：

| 建筑级字段 | 计算方式 |
|---|---|
| `LatestDataYear` | 最近有效年度 |
| `CurrentEUI` | 最近有效年度实际值 |
| `PredictedEUI` | 最近有效年度时间外预测 |
| `CurrentResidual` | 实际减预测 |
| `ResidualRatio` | 当前残差除以预测值 |
| `OfficialExcessRatio` | 当前实际值相对官方同类中位数超出比例 |
| `OfficialQuartile` | 当前值所在官方四分位 |
| `HighEnergyYears3Y` | 最近三年核心异常次数 |
| `ValidYears3Y` | 最近三年有效年份数 |
| `Persistence` | `HighEnergyYears3Y / ValidYears3Y` |
| `MedianResidual3Y` | 最近三年残差中位数 |
| `Trend3Y` | 至少两个有效点时的 `SiteEUI` 年度斜率 |
| `TheoreticalExcessEnergy_kBtu` | `max(CurrentResidual,0) × EUIReferenceArea` |
| `CurrentGHG` | 最近年度总 GHG |
| `CurrentGHGIntensity` | 最近年度 GHG 强度 |
| `Completeness3Y` | `ValidYears3Y / 3` |
| `ComplianceRate3Y` | 最近三年合规次数 / 记录次数 |
| `LatestIs2024` | 最近数据是否为 2024 |

理论超额能耗量单位换算：

```text
kBtu/year = kBtu/sf/year × sf
MMBtu/year = kBtu/year / 1,000
```

它只是“潜在节能规模代理指标”，不是实际可节省量。实际节能量还需要改造措施、运行工况、投资成本和工程可行性评估。

当 2024 缺失时使用最近有效年度，但必须保留 `LatestDataYear` 和 `LatestIs2024=False`；不得把旧数据伪装成当前状态。

核心候选建议要求：

```text
LatestDataYear >= 2023
HighEnergyYears3Y >= 2（有效年份不足 2 时不进入核心名单）
当前或最近有效年度同时满足模型异常和官方高耗能
数据质量合格
未拆除
```

## 14. 第十二步：优先级评分

### 14.1 先合并相关指标，再加权

残差、超耗比例和理论超额能耗都与能耗相关，不能分别以大权重重复计分。先组织成四个维度：

1. `InefficiencyScore`：相对低效程度；
2. `PersistenceScore`：持续高耗能；
3. `ExcessEnergyScore`：城市级理论超额能耗规模；
4. `CarbonScore`：碳减排意义。

默认原始分：

```text
RawPriority
  = 0.30 × InefficiencyScore
  + 0.20 × PersistenceScore
  + 0.25 × ExcessEnergyScore
  + 0.25 × CarbonScore
```

这些权重是透明的起始假设，不是客观真理，必须做敏感性分析。

### 14.2 归一化

使用百分位秩，避免极端值把 Min-Max 压缩到接近 0：

```python
def pct_rank(s):
    return s.rank(pct=True, method="average").fillna(0.0)

# 相对低效在同类内比较，保证类型公平
building["ResidualRatioPct"] = building.groupby("BenchmarkType")[
    "ResidualRatio"
].transform(pct_rank)
building["OfficialExcessPct"] = building.groupby("BenchmarkType")[
    "OfficialExcessRatio"
].transform(pct_rank)
building["InefficiencyScore"] = (
    building["ResidualRatioPct"] + building["OfficialExcessPct"]
) / 2

building["PersistenceScore"] = building["Persistence"].clip(0, 1)

# 城市总节能和总减排规模应跨类型比较
building["ExcessEnergyScore"] = pct_rank(
    np.log1p(building["TheoreticalExcessEnergy_kBtu"].clip(lower=0))
)
building["GHGTotalPct"] = pct_rank(
    np.log1p(building["CurrentGHG"].clip(lower=0))
)
building["GHGIntensityPct"] = pct_rank(
    building["CurrentGHGIntensity"].clip(lower=0)
)
building["CarbonScore"] = (
    building["GHGTotalPct"] + building["GHGIntensityPct"]
) / 2
```

评分前计算四维指标的 Spearman 相关性。如果某两维相关系数长期高于 0.85，应合并指标或降低重复权重，并在报告中解释。

### 14.3 数据质量作为惩罚系数

先在最近三年定义：

```text
ConsistencyRate3Y = 通过核心逻辑/算术检查的记录数 / 最近三年已有记录数
Currentness = 1.0（最新为 2024），0.7（最新为 2023），否则为 0
```

若最近三年没有记录，对应比率记为 0。`ConsistencyRate3Y` 的核心检查至少包括：面积有效、建造年份有效、GFA 恒等式成立、未命中人工确认的严重单位/上报错误。先用严重错误做硬排除，再对剩余建筑计算软质量分：

```text
DataQualityScore
  = 0.40 × Completeness3Y
  + 0.30 × ComplianceRate3Y
  + 0.20 × ConsistencyRate3Y
  + 0.10 × Currentness

FinalPriorityScore = 100 × RawPriority × DataQualityScore
```

存在严重单位/上报疑点的建筑不参与正常排序，单独进入核查名单；不要仅用一个较低质量分把明显错误数据继续留在 Top-N。

最终使用稳定次级键排序，保证相同得分可复现：先按 `FinalPriorityScore` 降序，再按理论超额能耗量降序，最后按 `OSEBuildingID` 升序。

### 14.4 候选原因必须可读

为每栋候选生成 `CandidateReason`，例如：

```text
2022—2024 有 3 年进入同类残差前 10%；
2024 SiteEUI 高于官方同类 75% 分位；
理论超额能耗量位于全市前 5%；
GHG 总量位于全市前 10%；
最近三年数据完整且合规。
```

最终候选表至少包含：

```text
Rank, OSEBuildingID, BuildingName, Address, Latitude, Longitude,
BenchmarkType, EPAPropertyType, LatestDataYear,
CurrentEUI, PredictedEUI, CurrentResidual, ResidualRatio,
OfficialMedianEUI, OfficialP75EUI, OfficialExcessRatio,
HighEnergyYears3Y, Persistence, Trend3Y,
TheoreticalExcessEnergy_kBtu, CurrentGHG, CurrentGHGIntensity,
DataQualityScore, FinalPriorityScore, CandidateReason
```

## 15. 第十三步：权重敏感性和排序稳定性

### 15.1 四套权重场景

| 场景 | 低效 | 持续 | 超额能耗 | 碳 |
|---|---:|---:|---:|---:|
| 均衡 | 0.30 | 0.20 | 0.25 | 0.25 |
| 节能规模优先 | 0.20 | 0.15 | 0.45 | 0.20 |
| 碳减排优先 | 0.20 | 0.15 | 0.20 | 0.45 |
| 持续异常优先 | 0.20 | 0.45 | 0.20 | 0.15 |

### 15.2 必算稳定性指标

- Top-20、Top-50、Top-100 重合数量；
- Jaccard 相似度；
- 全名单 Spearman 等级相关；
- Kendall 等级相关；
- 各 `BenchmarkType` 在名单中的占比；
- 每栋建筑在不同方案中的入选频率。

```text
Jaccard(A,B) = |A ∩ B| / |A ∪ B|
```

再进行按建筑 ID 的 bootstrap，重复整个聚合和排序流程，统计每栋建筑进入 Top-50 的比例。建议区分：

```text
StableCore：入选频率 ≥ 80%
WeightSensitive：40%—80%
Unstable：< 40%
```

由于没有真实改造完成后的标签，不能计算所谓“改造推荐准确率”。排序只能通过跨权重稳定性、数据质量、同类基准一致性和专家审阅进行间接验证。

阶段输出：

```text
outputs/tables/priority_all_buildings.csv
outputs/tables/priority_top20.csv
outputs/tables/priority_top50.csv
outputs/tables/priority_top100.csv
outputs/tables/ranking_sensitivity.csv
outputs/tables/topk_jaccard.csv
outputs/tables/bootstrap_inclusion_frequency.csv
```

## 16. 第十四步：对高优先级建筑做聚类画像

### 16.1 聚类对象

只对核心候选或优先级前 20% 建筑聚类。聚类用于回答“候选建筑有哪些典型画像”，不用于再次决定谁高耗能。

### 16.2 聚类特征

建议使用：

```text
log1p(PropertyGFABuildings)
BuildingAge
ResidualRatio
OfficialExcessRatio
Persistence
log1p(TheoreticalExcessEnergy_kBtu)
CurrentGHGIntensity
ParkingRatio
```

先检查高相关特征并删除冗余。连续变量使用 `RobustScaler`；极端规模变量先 `log1p`。建筑用途不经 `LabelEncoder` 后放入 K-Means，类别只用于聚类完成后的画像解释。

### 16.3 模型选择

核心使用 K-Means，比较 `k=2...8`；可补充层次聚类或 Gaussian Mixture。综合判断：

- Silhouette 越高越好；
- Calinski-Harabasz 越高越好；
- Davies-Bouldin 越低越好；
- 不同随机种子下 Adjusted Rand Index；
- 每群样本量是否合理；
- 画像是否有业务解释。

不能只根据肘部图机械选 k。

### 16.4 聚类命名和建议

按各群特征中位数、用途构成和典型建筑命名，例如：

- 大型、持续超耗、高排放办公建筑；
- 高强度酒店和医疗建筑；
- 老旧、中等规模、长期异常住宅；
- 小面积极端值、优先数据核查群。

措施建议只能写成审计方向，例如 HVAC 控制、围护结构、运行时段、蒸汽/燃气系统、照明和数据计量核查；不得根据聚类直接声称某项改造会产生确定节能率。

阶段输出：

```text
outputs/tables/candidate_clusters.csv
outputs/tables/cluster_profiles.csv
outputs/figures/cluster_metrics.png
outputs/figures/cluster_pca_projection.png
outputs/figures/cluster_profile_heatmap.png
```

## 23. 常见错误及正确处理

| 错误做法 | 正确做法 |
|---|---|
| 把 34,699 行当作 34,699 栋建筑 | 按 `OSEBuildingID` 识别 3,823 栋建筑 |
| 随机拆分全部行 | 按完整年份做时间外评估，并补充 ID 组划分 |
| 直接按主表 `BuildingType` 连接区间表 | 先构造 23 类 `BenchmarkType` |
| 用 LabelEncoder 编码用途 | One-Hot 或原生类别特征 |
| 将当年总能耗、GHG、ENERGY STAR 入模 | 仅用于 EDA、外部校验和排序 |
| 对 `SiteEUI` 直接 1%/99% 截尾 | 保留原始目标，极端值进入质量复核，模型用 `log1p`/鲁棒方法 |
| 用训练集内残差识别异常 | 使用滚动时间外预测残差 |
| `Residual > 0` 就判异常 | 同类前 10% + 官方 P75 + 数据质量 + 持续性 |
| 按绝对 EUI 全市排序 | 先做用途/类型公平校正，再考虑城市级规模 |
| 把超额 EUI × 面积称为实际节能量 | 称为理论超额能耗量或代理指标 |
| 给相关能耗指标重复大权重 | 先合成低效/规模/碳/持续四个维度 |
| 在所有建筑上机械聚类 | 只对高优先级候选做画像聚类 |
| 用 SHAP 声称因果关系 | 只解释模型关联和预测贡献 |
